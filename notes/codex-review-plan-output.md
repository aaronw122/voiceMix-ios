## Blocking issues

1. `integration-plan.md:39-42`, `MessagesExtension/VoiceTransformView.swift:15-40`, `MessagesExtension/VoiceTransformView.swift:544-553`: the plan says to ship the 3 live elevenlabs tiles now and add Trump/Obama later, but the implementation step still says to populate all 5 tiles in the new hardcoded catalog. If Trump/Obama land in the iOS catalog before the backend `voices.py` entries, tapping those tiles will call `/impersonate` with unknown `voiceId` and get the documented 404. Concrete fix: split the catalog step into two phases: phase 1 ships only `old-man`, `young-woman`, `femme-fatale`; phase 2 adds `trump` and `obama` in the same coordinated release as the backend entries, or gates them behind a remote `/voices` response.

2. `integration-plan.md:55`, `voiceMixer.xcodeproj/project.pbxproj:80-95`, `voiceMixer.xcodeproj/project.pbxproj:226-238`: creating `MessagesExtension/VoiceCatalog.swift` is not enough in this project. The existing project file explicitly lists Swift files in the MessagesExtension group and sources build phase, and there is no `VoiceCatalog.swift` reference today. Concrete fix: make the plan include adding `VoiceCatalog.swift` to the `voiceMixerMessages` target's `PBXFileReference`, MessagesExtension group, `PBXBuildFile`, and `PBXSourcesBuildPhase`; otherwise the extracted `VoicePersona` may not compile into the extension.

3. `integration-plan.md:56-64`, `MessagesExtension/VoiceTransformView.swift:5-40`, `MessagesExtension/VoiceTransformView.swift:297`: the persona-id to backend-voice-id migration is under-scoped. The current `VoicePersona.id` is both the SwiftUI `Identifiable` identity and the value sent to the service (`elder`, `aria`, `mlk`), while the backend expects `old-man`, `young-woman`, and `femme-fatale`. Adding `voiceId` is correct, but the plan only calls out one in-view edit. Concrete fix: define `VoicePersona.id` semantics explicitly, preferably keep `id` as a stable UI identity only if needed and add `voiceId` for network calls; then audit every use of `.id`, equality, `ForEach`, page dots, and tests so only `prepareClip` sends `voiceId`.

4. `integration-plan.md:66-73`, `MessagesExtension/LiveConvertService.swift:15-36`, `MessagesExtension/ConvertService.swift:20-24`, `MessagesExtension/VoiceTransformView.swift:286-291`: the plan flips `useMock = false` without requiring typed HTTP error handling. Today every non-2xx is collapsed into `invalidResponse`, and the UI shows only `Convert failed`. For the documented backend contract, 404, 413, 422, and 502 mean different actionable failures; during a real demo, a missing Trump/Obama voice, wrong endpoint/engine, too-large file, cold-start/proxy failure, and network outage will all look identical. Concrete fix: add `ConvertServiceError.httpStatus(Int, body: String?)` or domain cases for 404/413/422/502/network timeout; decode/log the response body; map statuses to specific UI status lines before enabling live mode.

5. `integration-plan.md:105-110`, `MessagesExtension/Config.swift:24-25`, `MessagesExtension/MessagesViewController.swift:9-10`: verification is too weak for the actual live switch. The plan says build and test elevenlabs voices, but it does not require verifying that the extension is actually using `LiveConvertService` after `useMock` flips. Concrete fix: add a pre-demo assertion/log that `Config.useMock == false` and `Config.baseURL == https://voiceapi.awill.co`, then validate real network traffic or backend logs for each shipped voice.

## Should-fix

1. `integration-plan.md:78-81`, `MessagesExtension/Config.swift:11-25`: hardcoding `baseURL` to production and `useMock = false` removes the current Info.plist/build-setting override path. The code already reads `API_BASE_URL` before falling back, and `steel.md` intended environment config for the extension target. Concrete fix: keep the Info.plist override and set the fallback/public value to `https://voiceapi.awill.co`, or add per-configuration build settings for `API_BASE_URL`; do not make production URL and live networking a manual source edit.

2. `integration-plan.md:79-80`, `MessagesExtension/Info.plist:1-32`, `App/Info.plist:1-38`: "HTTPS means no ATS exception" is true for the stated public origin, but the plan is silent on ensuring there is no stale `NSAppTransportSecurity` exception and on what happens when testing a plain HTTP local/tunnel endpoint. Concrete fix: add a checklist item to inspect both plists for ATS overrides before release, and document that HTTP dev endpoints require a debug-only extension-target ATS exception.

3. `integration-plan.md:72-73`, `MessagesExtension/LiveConvertService.swift:50-79`: `/impersonate` accepts exactly one of `audio` or `text`; the plan correctly uses audio only, but it says the multipart body is identical and does not require a test proving only one payload part is emitted. Concrete fix: add unit coverage or a request-builder check for modal voices confirming fields are exactly `voiceId` + `audio`, with no `text` part.

4. `integration-plan.md:96-101`: treating `/voices` drift checking as optional is risky because the endpoint exists and the current plan depends on a hardcoded engine field. A backend-side engine change or removed voice becomes a 422/404 at demo time. Concrete fix: move the `/voices` fetch into the core plan at least as a DEBUG preflight, and fail loudly if the local catalog's `voiceId`, `engine`, or `acceptsText` no longer matches.

5. `integration-plan.md:90-91`, `MessagesExtension/AudioRecorder.swift:8`, `MessagesExtension/AudioRecorder.swift:26-31`, `MessagesExtension/LiveConvertService.swift:23-31`: the 60-second cap aligns with `MAX_SECONDS`, but "recordings are well under 10 MB" is an assumption, not a guard. The live service currently loads the entire file and multipart body into memory before upload. Concrete fix: check file size before conversion and return a specific "too long/too large" error before reading it into memory; longer-term, use streaming upload/download or at least avoid duplicate buffering in the extension.

6. `integration-plan.md:105-110`, `MessagesExtension/VoiceTransformView.swift:82-87`, `MessagesExtension/VoiceTransformView.swift:346-353`: modal cold starts of 15-45s are called out in the prompt but not handled by the plan. The current status copy reaches "Rendering new audio..." after a few seconds and then stalls indefinitely. Concrete fix: add timeout policy and cold-start-specific status behavior before wiring modal voices; for phase 1 elevenlabs only, explicitly defer modal UX until JFK/Trump/Obama are tested.

7. `MessagesExtension/LiveConvertService.swift:72-73`: the multipart filename is inserted directly into the `Content-Disposition` header. It currently comes from a generated temp file, so the risk is low, but it is still a header-construction footgun if a future file source has quotes or CRLF. Concrete fix: sanitize or fixed-name the upload filename, e.g. `recording.m4a`.

8. `MessagesExtension/MockConvertService.swift:9-17`, `integration-plan.md:76`: mock/live parity should include endpoint-routing behavior, not just signature parity. If the mock ignores `engine`, tests can still pass while live modal/elevenlabs routing is wrong. Concrete fix: make the mock optionally assert known voice/engine pairings in DEBUG or expose a test spy service that records `voiceId` and `engine`.

## Verdict

The plan is not sound enough to implement as written. The endpoint/engine model matches the backend contract in principle, and the existing code has a clean service seam, but the plan has release-blocking holes around the hardcoded catalog split, the new Xcode source file, typed live error handling, and worktree coordination with an actively changing `VoiceTransformView.swift`. Fix those items first, then the "ship the 3 elevenlabs voices now, add modal voices later" split becomes implementable and much less likely to fail during a real demo.
