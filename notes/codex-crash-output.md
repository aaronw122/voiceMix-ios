# iMessage Extension Crash Investigation

## Ranked hypotheses

| Rank | cause | why it fits | how to check |
|---:|---|---|---|
| 1 | **Extension memory pressure / jetsam during audio decode or export** | **Likely.** The observed `SIGKILL(9)` with no Swift backtrace and no `.ips` crash report is exactly what a system kill can look like. Apple documents that visible apps can appear to crash when jettisoned for memory pressure, and that app extensions have a much lower per-process memory limit than foreground apps. I did not find an official current iMessage-specific number; community reports often cite share/extension budgets around tens of MB to ~120 MB, but treat those as device/extension-class dependent, not a contract. This code now echoes the user's real recording, so `WaveformVideoRenderer.readPCMAmplitudes` stores every decoded sample in `[CGFloat]` before reducing to 40 bars. At 44.1 kHz mono, that is about 2.65M `CGFloat`s per minute, roughly 21 MB/min just for the array on 64-bit, before `Data` chunks, AVAssetReader, image buffers, AVAssetWriter, composition/export allocations, and Messages host overhead. Longer clips can plausibly cross an extension budget. | Attach Xcode to the **MessagesExtension process**, not only Messages. Use Debug Navigator memory gauge while recording 2s, 10s, 30s, 60s clips. Also collect device logs after a kill: in Console.app search for the extension bundle id plus `jetsam`, `memorystatus`, `per-process-limit`, or `highwater`. On device, pull analytics/jetsam reports from Xcode Organizer / Devices and Simulators if present. Temporarily make `waveformBars` return `[]` and retest; if kills disappear or memory drops sharply, the PCM array is confirmed. |
| 2 | **Long-running encode/export exceeds extension presentation/watchdog expectations** | **Likely.** The console says `Remote app card controller ... timed out during delayed presentation`, followed by plugin interruption/invalidated messages. Even if the main actor is mostly free, Messages is hosting this as a remote extension UI and may have strict expectations while the card is expanding/presenting. The regression from a 2s bundled sample to a real recording maps to a much longer `writeVideoTrack` plus `AVAssetExportSession` lifecycle. | Add `os_log` or `os_signpost` around: stop recording, `convert`, `fetchAudio`, waveform read start/end, `writeVideoTrack` start/end, `mux` start/end, export completion/error. Verify whether the kill happens in waveform decode, video write, or export. Time each step for 2s/10s/30s/60s. Temporarily cap duration to 5-10 seconds and see whether the timeout disappears. |
| 3 | **Two-pass video construction is too expensive for an extension** | **Likely.** The renderer first writes a video-only H.264 file for the full duration, then creates an `AVMutableComposition` and uses `AVAssetExportSession` with `AVAssetExportPresetMediumQuality` to mux/re-encode to mp4. That doubles I/O and invokes a second media pipeline. For a static cover, this is more work than necessary. | Stub `makeVideo` to return the audio URL directly and change `insertAttachment` to attach the raw `.m4a`; if the extension survives, the record/convert plumbing is not the killer. Next, keep `makeVideo` but skip waveform generation. Then replace the final `AVAssetExportSession` with a single `AVAssetWriter` that writes one video track plus an AAC audio track from `AVAssetReader`. Compare memory and wall-clock. |
| 4 | **AVFoundation export failure or hang around `err=-12900`** | **Plausible.** The lone CoreMedia/Fig `err=-12900` could be an export/asset pipeline failure, especially with an invalid time range, unsupported track format, simulator media services issue, or composition/export problem. The code inserts `duration` from the original audio into both video and audio tracks. If the video-only asset's actual duration is shorter than requested, `insertTimeRange` or export can fail. In the iOS 18 path, `export.outputURL`/`outputFileType` are set and then `export.export(to:as:)` is also called; that is probably redundant rather than fatal, but it should be verified on the target OS. | Log `asset.duration`, `videoAsset.duration`, track durations, `export.status`, `export.error`, and `export.progress`. Add a timeout wrapper around export and call `export.cancelExport()` on timeout. Reproduce with known short `.m4a`, longer `.m4a`, and the old bundled `.mp3`. Run on physical device and simulator separately. |
| 5 | **Main actor / presentation blocking** | **Possible but lower than memory/time.** The important nuance: `MessagesViewController` is effectively main-actor isolated through UIKit, and `prepareClip` is an instance method, so its own pre/post-await code is main-actor isolated. However `service.convert`, `service.fetchAudio`, and `WaveformVideoRenderer.makeVideo` are nonisolated async methods on non-actor types, so their bodies should execute on the generic cooperative executor after suspension, not continuously on the main actor. The heavy renderer body is therefore probably not pinned to the main actor. Still, UIKit drawing APIs inside `makeCoverImage`/`drawWaveform` are being used off-main; that may be okay for image-context drawing in practice, but it is worth checking with Thread Sanitizer / Main Thread Checker. | Add `Thread.isMainThread` logs at `prepareClip`, `MockConvertService.convert`, `fetchAudio`, `WaveformVideoRenderer.makeVideo`, `readPCMAmplitudes`, `writeVideoTrack`, and `mux`. Use Time Profiler to see whether the main thread is blocked during conversion. Enable Main Thread Checker. |
| 6 | **`requestPresentationStyle(.expanded)` timing / delayed presentation interaction** | **Possible contributor.** `beginRecording` requests expanded presentation immediately before activating the audio session and starting `AVAudioRecorder`. The kill happens after Stop/Convert per the symptom, so this is unlikely to be the root cause by itself. But Messages may still be in a delayed presentation state when a long conversion starts. | Test three variants: do not request expanded at all; request expanded in `viewDidAppear`; request expanded before recording and delay conversion until `didTransition(to:)` confirms expanded. If only one variant triggers `timed out during delayed presentation`, the presentation lifecycle is involved. |
| 7 | **AVAudioSession activation/deactivation in an iMessage extension** | **Possible but lower.** `AudioRecorder.startRecording` sets `.playAndRecord` and activates the shared session. `stopRecording` deactivates it before video rendering. The Info.plist has `NSMicrophoneUsageDescription`, so this is not the known privacy hard-crash case. AVAudioSession problems would more likely fail recording or log audio-session errors than kill during encode. | Log errors from `setCategory`, `setActive`, and `record()`. Keep the session active through conversion in one test; in another, skip conversion after stop and immediately show Send for the raw audio. If the kill only occurs with rendering, audio session is not primary. |
| 8 | **Simulator host degradation / Messages environment** | **Plausible environmental amplifier.** The brief notes simulator Messages/iCloud/XPC problems. That can produce plugin invalidation noise and media-services instability. But the same failure correlated with longer real recordings is code-specific enough that simulator degradation should not be the lead hypothesis. | Reproduce on a physical device with a clean install and device Console open. Also reset simulator content/settings or use a fresh simulator runtime. If physical device only fails for longer clips, fix renderer resource usage. If simulator fails for everything while device passes, treat simulator logs as contaminated. |

## Code bugs found

- `MessagesExtension/WaveformVideoRenderer.swift:193`: **High-risk memory bug.** `readPCMAmplitudes` materializes one `CGFloat` per decoded PCM sample, then `normalizedBars` reduces that huge array to 40 bars. For extension code, this is the clearest concrete resource bug. Stream directly into 40 buckets instead.

- `MessagesExtension/WaveformVideoRenderer.swift:201`: **Autorelease / transient allocation risk.** The loop creates a new `Data(count: length)` for every sample buffer and copies the block buffer before iterating. Those allocations are temporary, but during long reads they still increase pressure. Use `CMBlockBufferAccessDataBytes` where possible, or process smaller buffers inside `autoreleasepool`.

- `MessagesExtension/WaveformVideoRenderer.swift:329`: **Unbounded work scales with duration.** `staticFrameTimes` returns 6 fps for the full audio duration, then the code appends the same pixel buffer repeatedly. For a static image, this produces hundreds/thousands of frames for long clips. Low fps helps file size but does not eliminate encoder wall-clock.

- `MessagesExtension/WaveformVideoRenderer.swift:333`: **Append result is ignored.** `adaptor.append(buffer, withPresentationTime:)` returns `Bool`. If it fails, the code continues, marks finished, and only discovers writer failure later. Log/throw immediately with `writer.error`.

- `MessagesExtension/WaveformVideoRenderer.swift:330`: **Potential hang/spin on writer backpressure.** The `while !input.isReadyForMoreMediaData { sleep 5ms }` loop has no cancellation check, no timeout, and no writer status check. If the writer fails or stalls, the task can sit there until the extension is killed.

- `MessagesExtension/WaveformVideoRenderer.swift:338`: **No cancellation cleanup.** If the task is cancelled or an earlier append fails, `writer.cancelWriting()` is never called. Similarly, if `mux` throws, `videoOnlyURL` is not removed because cleanup only happens after a successful mux at line 290.

- `MessagesExtension/WaveformVideoRenderer.swift:347`: **Duration mismatch edge.** The final frame time is `(frameCount - 1) / fps`, but `writer.endSession(atSourceTime: duration)` asks the session to end at the audio duration. This is probably okay, but for some durations the resulting video asset can be slightly shorter than the audio range later inserted at line 379. Log actual `videoAsset.duration` before muxing.

- `MessagesExtension/WaveformVideoRenderer.swift:390`: **Medium-quality export may re-encode more than needed.** `AVAssetExportPresetMediumQuality` can be excessive for a static 600x600 cover and a voice clip. It also forces a second media pipeline after the video-only writer.

- `MessagesExtension/WaveformVideoRenderer.swift:398`: **iOS 18 export path should still log final status/error.** `try await export.export(to:as:)` should throw on failure, but logging `status`, `error`, and duration after return will make `err=-12900` diagnosable. The existing legacy path throws a string but loses detailed error domain/code unless localizedDescription includes them.

- `MessagesExtension/MessagesViewController.swift:114`: **Lifecycle coupling.** Conversion starts inside the live Messages extension UI immediately after Stop. If encode takes several seconds or spikes memory, the extension has no protection from presentation/watchdog pressure. There is also no timeout/cancel path if the user dismisses the extension.

- `MessagesExtension/MockConvertService.swift:20`: **Regression source confirmed.** The mock now copies and returns the actual recorded file. That means all downstream rendering cost scales with user recording length instead of the previous tiny bundled `sample.mp3`.

## Recommended minimal fix

1. **First verification patch:** temporarily bypass `WaveformVideoRenderer.makeVideo` and insert the raw `.m4a` or old `sample.mp3`. If the kill disappears, the renderer is confirmed.

2. **Add instrumentation before refactoring:** bracket `waveformBars`, `writeVideoTrack`, and `mux` with `Logger`/`os_signpost`; log duration, file size, thread, memory warnings, export status/error, and elapsed time. This prevents guessing between memory, writer, and export.

3. **Replace per-sample storage with streamed 40-bucket waveform.** Do not build `[CGFloat]` for the whole recording. Compute bucket sums/counts while reading PCM, or skip waveform generation entirely for the first fix and use the static mic cover. This is the smallest likely memory win.

4. **Cap recording/render duration in the extension.** For example, enforce 10-15 seconds while proving the flow. If product needs longer clips, do not encode a long mp4 inside the iMessage extension lifecycle.

5. **Make writer loops fail fast.** Check `Task.isCancelled`, `writer.status`, and the return value of `adaptor.append`. Add a timeout around both writer finish and export; call `cancelWriting()` / `cancelExport()` and delete temp files on failure.

6. **Reduce video work.** Drop frame size to 320x320 or 400x400 and fps to 1-2 for the static-cover path. Use lower bitrate compression properties if keeping H.264. The current 600x600 at 6 fps is more than a voice bubble cover needs.

7. **Prefer a single-pass writer if mp4 is mandatory.** The more robust design is `AVAssetReader` for source audio plus `AVAssetWriter` with one static video track and one AAC audio track, avoiding the video-only intermediate file and `AVAssetExportSession` composition pass.

8. **Decouple long work from presentation if longer clips are required.** For a steel-thread iMessage extension, keep conversion/rendering short and synchronous enough to finish while the card is alive. For production-length audio, upload to backend and return a small already-encoded asset, or hand work to the containing app / background-capable path and let the extension only insert completed media.

## Certain vs uncertain

- **Certain from code:** the mock regression now feeds real recording duration into the renderer; the renderer reads all PCM amplitudes into memory; the renderer does a video-only write followed by export/mux; append/backpressure/export paths lack robust timeout/cancellation/logging.

- **Likely from symptoms:** `SIGKILL(9)` with no Swift backtrace points more toward system termination such as jetsam/watchdog than a Swift exception.

- **Uncertain:** the exact iMessage extension memory budget on modern iOS. Apple documents lower extension limits and jetsam diagnosis, but does not appear to publish a stable iMessage-specific MB number. Measure the actual process on the target device.
